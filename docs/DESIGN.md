# WCamper Webtoon 설계 문서

## 목적

WCAMPER Webtoon은 초기에 많은 카테고리를 나누기보다, 하나의 대표 작품을 중심으로 독자가 바로 읽고 반응할 수 있는 웹툰 서비스형 사이트로 구성한다. 첫 MVP는 정적 게시 사이트이지만, 이후 운영에서는 방문, 조회, 완독, 재방문, 피드백 데이터가 백그라운드에서 축적되어 작가, 작품, 회차 스코어링으로 연결되어야 한다.

운영자는 작가, 작품, 회차 단위로 목표를 설정하고, 축적된 행동/피드백 데이터로 목표 달성 여부를 판정한다. 목표 달성 결과는 연재 지속, 다음 회차 제작 우선순위, 작가/작품 신뢰 뱃지, 내부 스코어링 보정에 사용한다.

## 제품 구성

### 독자 홈

- 대표 작품: `BD크루 캠핑일지`
- 최신 회차
- 작품 소개
- 회차 목록
- 등장 크루
- 작가 프로필
- 작가의 다른 작품, 연재 준비작, 기획작
- 제작 노트
- 독자 피드백
- 목표와 달성

초기에는 `요일별`, `완결`, `도전` 같은 대분류를 먼저 열지 않는다. 콘텐츠가 충분해질 때까지는 작품 하나를 중심으로 사이트를 채우고, 독자가 읽기, 관심 등록, 피드백 남기기까지 자연스럽게 이어지는 흐름을 우선한다.

### 작가 영역

작가 영역은 단순 소개가 아니라 작가 단위 신뢰와 선호도를 쌓는 단위다.

- 작가명
- 작가/스튜디오 소개
- 주요 키워드
- 데뷔 또는 파일럿 정보
- 대표 작품
- 연재 중 작품
- 준비 중 작품
- 기획 중 작품
- 작가 대상 피드백
- 작가 스코어

### 작가 제작/업로드 영역

작가는 웹툰을 직접 제작하고 플랫폼에 제출할 수 있어야 한다. 초기 모델은 플랫폼 안에 완성형 저작도구를 먼저 넣기보다, 작가가 GPT, 이미지 생성 도구, 로컬 편집 도구 등 자신이 쓰는 AI 제작 도구에서 만든 결과물을 WCAMPER Webtoon에 올릴 수 있게 하는 API 중심 구조로 둔다.

초기 작가 업로드는 다음 흐름을 기준으로 한다.

1. 작가가 `auth.wcamper.com`으로 인증한다.
2. 작가 또는 작가가 사용하는 AI 도구가 작품/회차 초안을 생성한다.
3. API, MCP, GPT Actions 중 하나로 회차 패키지를 제출한다.
4. 서버는 파일 무결성, 필수 메타데이터, 저작권/출처 고지, 안전성 기준을 검사한다.
5. 제출본은 즉시 공개하지 않고 `draft` 또는 `review` 상태로 저장한다.
6. 운영자 또는 자동 검수 작업이 승인한 뒤 공개 카탈로그에 반영한다.

작가 업로드 영역은 독자 피드백/스코어링과 연결되지만, 제출 권한과 공개 권한은 분리한다. 인증된 작가는 초안을 만들고 제출할 수 있으며, 공개 전환은 운영 정책에 따른다.

### 작품 영역

작품은 독자가 관심 등록하고 계속 따라가는 핵심 단위다.

- 작품명
- 작가
- 연재 상태
- 장르 태그
- 한 줄 소개
- 커버
- 회차 목록
- 조회, 좋아요, 관심 수
- 작품 대상 피드백
- 작품 스코어

### 회차 영역

회차는 실제 읽기 경험과 가장 가까운 평가 단위다.

- 회차 번호
- 회차 제목
- 공개일
- 공개 상태
- 썸네일
- 읽는 시간
- 컷 목록
- 회차별 조회 수
- 회차별 완독률
- 회차별 좋아요
- 회차 대상 피드백
- 회차 스코어

## 작가 제출 API 설계

작가 제출 API는 외부 AI 도구와 플랫폼 사이의 안정적인 계약이다. 목표는 특정 AI 도구에 종속되지 않고, GPT Actions, MCP 클라이언트, 자체 제작 worker, 로컬 스크립트가 같은 제출 규격을 사용할 수 있게 하는 것이다.

### 제출 패키지

회차 제출은 메타데이터와 컷 자산을 하나의 패키지로 다룬다.

```json
{
  "submissionId": "nullable-before-create",
  "authorId": "author-bd-crew-studio",
  "seriesId": "bd-crew-camping-log",
  "episode": {
    "episodeNo": 2,
    "title": "비 오는 날의 타프",
    "summary": "비 오는 캠핑장에서 BD크루가 첫 장비 문제를 해결한다.",
    "language": "ko",
    "ageRating": "all",
    "status": "draft"
  },
  "assets": [
    {
      "assetType": "panel_image",
      "panelNo": 1,
      "fileName": "panel-001.png",
      "mimeType": "image/png",
      "sha256": "hex-encoded-hash",
      "width": 1440,
      "height": 2200
    }
  ],
  "creationDisclosure": {
    "usedAiTools": true,
    "tools": ["GPT", "image-generation-tool"],
    "humanEdited": true,
    "sourceMaterialPolicy": "author-owned-or-licensed"
  }
}
```

필수 검증 기준은 다음과 같다.

- 인증된 작가 또는 위임된 도구만 제출 가능
- 작가가 소유한 작품 또는 제출 권한이 있는 작품만 수정 가능
- 컷 순서, 파일 해시, 이미지 크기, MIME 타입 확인
- AI 도구 사용 여부와 인간 편집 여부 고지
- 외부 저작물, 실존 인물, 민감 소재 사용 시 추가 검수 플래그 기록
- 제출본은 기본적으로 `draft` 또는 `review` 상태이며 바로 공개하지 않음

### API 후보

| Method | Path | 목적 |
|---|---|---|
| `POST` | `/api/author/submissions` | 작품/회차 제출 초안 생성 |
| `POST` | `/api/author/submissions/:submissionId/assets` | 컷 이미지, 썸네일, 원고 파일 업로드 URL 발급 또는 자산 등록 |
| `PATCH` | `/api/author/submissions/:submissionId` | 제목, 설명, 컷 순서, 공개 희망일 등 메타데이터 수정 |
| `POST` | `/api/author/submissions/:submissionId/submit-review` | 운영 검수 요청 |
| `GET` | `/api/author/submissions` | 작가 본인의 제출 목록 조회 |
| `GET` | `/api/author/submissions/:submissionId` | 제출 상세와 검수 상태 조회 |
| `POST` | `/api/author/series` | 작가가 새 작품 초안을 생성 |
| `GET` | `/api/author/series` | 작가 본인의 작품 목록 조회 |
| `GET` | `/api/author/capabilities` | 현재 계정의 제출 권한, 허용 파일 형식, 용량 제한 조회 |
| `POST` | `/api/moderation/submissions/:submissionId/approve` | 운영자 공개 승인 |
| `POST` | `/api/moderation/submissions/:submissionId/reject` | 운영자 반려 및 사유 기록 |

초기 구현에서는 업로드 바이너리를 API 서버가 직접 받기보다, 사전 서명 업로드 URL 또는 별도 자산 저장소를 사용하는 방식을 우선 검토한다. API 응답이나 로그에는 토큰, 원본 인증 정보, 비공개 파일 URL을 남기지 않는다.

## AI 도구 연동: MCP와 GPT Actions

초기 작가 경험은 웹 대시보드보다 AI 도구 연동을 먼저 제공하는 방향이 적합하다. 작가가 이미 쓰는 GPT, 로컬 에이전트, 이미지 생성 도구에서 회차 패키지를 만들고, 플랫폼은 표준화된 제출/검수 API를 제공한다.

### MCP 서버

MCP는 작가용 AI 클라이언트가 WCAMPER Webtoon의 작품 기준, 캐릭터 기준, 회차 제출 API를 안전하게 사용할 수 있게 하는 어댑터다.

초기 MCP tool 후보:

| Tool | 목적 |
|---|---|
| `get_author_profile` | 인증된 작가 프로필과 제출 가능 작품 조회 |
| `list_series` | 작가의 작품 목록 조회 |
| `get_series_guidelines` | 작품별 세계관, 캐릭터, 톤, 이미지 규격 조회 |
| `create_episode_draft` | 회차 초안 생성 |
| `upload_panel_asset` | 컷 자산 등록 또는 업로드 URL 요청 |
| `validate_submission` | 컷 누락, 해시 불일치, 필수 고지 누락 검사 |
| `submit_for_review` | 검수 요청 |
| `get_submission_status` | 제출/검수 상태 조회 |

MCP 서버는 작가 인증 토큰을 대화 로그나 산출물에 노출하지 않아야 하며, 권한 확인은 백엔드 API에서 최종 판정한다.

### GPT Actions

GPT Actions는 비개발자 작가가 GPT 안에서 회차 제출을 시작할 수 있게 하는 공개 OpenAPI 계약으로 본다.

초기 Actions 범위:

- 작가 권한 확인
- 작품/회차 초안 생성
- 제출 패키지 JSON 검증
- 업로드 URL 요청
- 검수 요청
- 검수 상태 조회

GPT Actions는 실제 이미지 바이너리 업로드보다 메타데이터 생성과 업로드 URL 발급에 초점을 둔다. 대용량 파일 전송, 이미지 리사이징, 바이러스 스캔, 저작권 검수는 백엔드 또는 별도 worker에서 처리한다.

## AI 웹툰작가 저작도구 Worker

완성형 AI 웹툰 저작도구는 본 플랫폼 안에 바로 섞기보다 별도 worker 구성으로 서비스화하는 것이 적절하다. 이유는 제작 워크플로우가 독자 서비스와 다른 리소스, 권한, 비용, 장애 격리를 필요로 하기 때문이다.

### 분리 이유

- 이미지 생성/편집은 긴 실행 시간과 높은 비용을 가진다.
- 모델 키, 프롬프트, 원본 제작 파일은 독자 서비스보다 더 강한 보안 경계가 필요하다.
- 제작 실패, 재시도, 큐 처리, 생성 이력 관리가 필요하다.
- 작가별 스타일 가이드와 캐릭터 일관성 검사가 별도 도메인 로직이 된다.
- 공개 서비스 장애와 제작 worker 장애를 분리할 수 있다.

### worker 책임

- 작품별 세계관/캐릭터/톤 기준 조회
- 원고, 콘티, 컷 분할, 대사 초안 생성 보조
- 이미지 생성/리터치 작업 큐 처리
- 컷 규격 변환, 썸네일 생성, 해시 계산
- 제작 이력과 AI 사용 고지 자동 구성
- 제출 API로 회차 패키지 전달

### 플랫폼 책임

- 인증과 작가 권한 판정
- 작품/회차/제출 상태 관리
- 자산 저장소 연결
- 검수 및 공개 승인
- 독자 조회, 피드백, 목표, 스코어링 집계

따라서 초기 단계는 `작가 제출 API + MCP + GPT Actions`를 먼저 만들고, 이후 별도 `AI 웹툰작가 저작도구 worker`가 같은 API를 사용하는 내부 클라이언트로 발전시키는 순서가 좋다.

## Vercel 기반 MVP/베타 운영 기준

장기 구조는 정적 사이트가 아니라 `Vercel 정적 프론트 + API + DB + 자산 저장소 + 별도 worker`로 본다. 다만 MVP와 제한된 베타에서는 비용을 늦추기 위해 무료 또는 저비용 구간에서 시작할 수 있다.

### 권장 초기 구조

```text
Vercel Static Frontend
+ Vercel Functions API
+ Neon 또는 Supabase Postgres
+ Vercel Blob, R2, S3 계열 object storage
+ 집계 중심 이벤트 저장
+ 별도 AI 웹툰작가 저작도구 worker
```

정적 프론트에는 공개 가능한 작품/회차 메타데이터와 캐시 가능한 읽기 화면만 둔다. 작가 제출, 피드백, 조회 이벤트, 목표, 스코어링, 검수 상태는 API와 DB에서 관리한다. 웹툰 이미지 원본과 업로드 자산은 Git 저장소나 DB에 넣지 않고 object storage에 저장한다.

### 2026-07-08 기준 무료 구간 참고치

무료 구간 수치는 벤더 정책 변경 가능성이 있으므로 베타 시작 전 다시 확인한다.

| 영역 | 무료 기준 참고치 | 설계상 의미 |
|---|---:|---|
| Vercel Functions | 월 1,000,000 invocations, Active CPU 월 4 CPU-hours | API 호출은 가능하지만 이벤트를 요청마다 무겁게 처리하면 빠르게 한계에 닿음 |
| Vercel Fast Data Transfer | 월 100GB | 웹툰 회차 이미지 전송량이 핵심 병목 |
| Vercel Blob | 저장 1GB, 전송 10GB, simple operations 10,000, advanced operations 2,000 | 작가 업로드 이미지의 실제 운영 저장소로는 매우 이른 시점에 부족 |
| Neon Free Postgres | 프로젝트당 storage 0.5GB, compute 100 CU-hours, public transfer 5GB | 메타데이터와 제한된 피드백은 가능하지만 raw event log 장기 보관에는 부족 |

### 무료 베타 가정치

다음 수준은 무료 또는 저비용 구간에서 검증 가능한 목표치로 둔다. 실제 가능량은 회차 이미지 평균 크기, 캐시 적중률, 이벤트 수집 방식에 따라 달라진다.

| 항목 | 무료 베타 목표 범위 |
|---|---:|
| 작가 | 5~20명 |
| 작품 | 20~100개 |
| 회차 | 100~500화 |
| 월 방문자 | 1,000~5,000명 |
| 월 회차 열람 | 회차당 이미지 용량에 따라 500~5,000회 |
| 원본 이벤트 로그 | 장기 보관하지 않고 집계 중심 |

무료 베타에서는 모든 이벤트를 원본 row로 오래 저장하지 않는다. `episode_view`, `panel_impression`, `episode_complete` 같은 고빈도 이벤트는 일/작품/회차 단위 집계 테이블을 먼저 갱신하고, 원본 이벤트는 샘플링하거나 짧은 보관 기간을 둔다.

### 유료 전환 트리거

다음 조건 중 하나가 발생하면 유료 전환 또는 별도 인프라 전환을 검토한다.

| 트리거 | 판단 |
|---|---|
| 이미지 전송량이 월 50~100GB에 접근 | 회차 이미지가 비용의 주 병목이므로 object storage/CDN 비용 구조를 확정 |
| Vercel Blob 저장 0.7~1GB 또는 전송 7~10GB 접근 | Blob 무료 한계 전 유료 저장소 또는 R2/S3 계열 전환 결정 |
| DB 크기 300~400MB 접근 | 0.5GB 무료 DB 한계 전에 이벤트 보관 정책, 인덱스, 유료 플랜 결정 |
| 월 API 호출 300,000~500,000회 초과 | Vercel Functions 한도 여유를 두고 CPU, DB 연결, 로그 비용 점검 |
| raw event log가 월 100,000~300,000 rows 이상 | 원본 로그 저장소와 집계 파이프라인 분리 |
| 외부 작가 공개 모집 또는 상업적 베타 시작 | Vercel Hobby의 개인/비상업 성격을 벗어나는지 확인하고 Pro 전환 검토 |
| 작가 업로드 이미지가 정기적으로 발생 | 업로드 URL, 스캔, 리사이징, 검수 큐, 보존 정책 필요 |

전환 시점의 1순위는 DB보다 이미지 자산과 이벤트 설계다. 작품/회차/작가 메타데이터는 무료 DB로도 오래 버틸 수 있지만, 웹툰 이미지는 저장량과 전송량이 동시에 증가하고, 이벤트 원문은 row 수가 빠르게 늘어난다.

### 비용 통제 원칙

- GitHub 또는 Git 저장소에는 샘플 이미지와 코드만 둔다.
- 이미지 원본, 컷, 썸네일은 object storage에 저장하고 CDN 캐시를 사용한다.
- API는 이미지 바이너리를 직접 프록시하지 않고 업로드 URL과 공개 자산 URL을 관리한다.
- 이벤트 원문은 짧게 보관하고 화면/스코어링은 집계 테이블을 사용한다.
- 월별로 `이미지 전송량`, `Blob/storage 사용량`, `DB size`, `API invocations`, `event rows`, `작가 제출량`을 확인한다.
- 무료 한도 70%에 도달하면 유료 전환 또는 구조 변경 결정을 시작한다.

## 통합인증

피드백 작성은 `auth.wcamper.com` 통합인증과 연결하고, 로그인 회원에게만 허용한다.

### 인증 원칙

- 익명 사용자는 읽기와 조회만 허용한다.
- 인증 회원은 피드백, 관심 등록, 반복 참여, 신고, 운영자 검수 이력과 연결한다.
- 인증 토큰, 세션, 개인 식별 정보는 프론트 정적 데이터나 공개 로그에 남기지 않는다.
- 스코어링에는 원본 개인정보가 아니라 익명화된 사용자 키와 집계 신호를 사용한다.

### 사용자 유형

| 유형 | 설명 | 스코어링 반영 |
|---|---|---|
| 익명 방문자 | 로그인하지 않은 방문자 | 조회/완독 집계만 반영, 피드백 작성 불가 |
| 인증 회원 | `auth.wcamper.com`으로 확인된 사용자 | 피드백 기본 가중치 |
| 신뢰 회원 | 반복 참여, 낮은 신고율, 운영자 검수 통과 이력이 있는 사용자 | 높은 가중치 |
| 운영자 | 검수, 숨김, 신고 처리 권한 보유 | 운영 판정 신호 |

## 피드백 설계

피드백은 하나의 댓글함이 아니라 대상별로 분리한다.

### 피드백 대상

| 대상 | 목적 | 예시 |
|---|---|---|
| 작가 | 작가의 세계관, 말투, 연재 방향 평가 | `BD크루 스튜디오의 톤이 좋은가` |
| 작품 | 작품 콘셉트, 장르, 계속 볼 의향 평가 | `BD크루 캠핑일지를 관심 등록할 만한가` |
| 회차 | 컷 흐름, 재미, 다음 화 기대감 평가 | `1화의 시작 몰입감이 있는가` |

### 피드백 종류

- 짧은 텍스트 피드백
- 좋아요 또는 공감
- 관심 등록
- 완독 후 반응
- 다음 화 기대
- 신고
- 운영자 검수 메모

### 인증 회원 피드백

인증 회원 피드백은 스코어링의 주 신호로 사용한다.

- `auth.wcamper.com` 세션 검증 필수
- `UserProfile.completedAt` 완료 회원만 작성 허용
- 사용자별 반복 참여 반영
- 장기 선호도 반영
- 신고 이력 반영
- 운영자 검수 결과 반영
- 관심 등록, 재방문, 완독과 연결

## 백그라운드 데이터 축적

사이트가 정적 MVP로 시작하더라도, 운영 설계상 백그라운드에서는 방문 및 조회 데이터가 계속 쌓여야 한다. 이 데이터는 피드백과 결합되어 작가, 작품, 회차의 점수와 추천 기준이 된다.

### 수집 이벤트

| 이벤트 | 설명 | 주요 대상 |
|---|---|---|
| `page_view` | 페이지 진입 | 사이트, 작품, 회차 |
| `series_view` | 작품 상세 또는 작품 홈 노출 | 작품 |
| `episode_view` | 회차 뷰어 진입 | 회차 |
| `panel_impression` | 컷이 화면에 노출됨 | 회차, 컷 |
| `episode_complete` | 마지막 컷 또는 하단 영역 도달 | 회차 |
| `favorite_added` | 관심 등록 | 작가, 작품 |
| `feedback_created` | 피드백 작성 | 작가, 작품, 회차 |
| `feedback_reacted` | 피드백 공감 또는 비공감 | 피드백 |
| `report_created` | 신고 | 피드백, 사용자 |
| `return_visit` | 재방문 | 사이트, 작품 |

### 이벤트 공통 필드

```json
{
  "eventId": "uuid",
  "eventType": "episode_view",
  "occurredAt": "2026-07-07T00:00:00.000Z",
  "sessionId": "browser-session-key",
  "userKey": "nullable-auth-user-key",
  "authState": "anonymous|member",
  "targetType": "author|series|episode|panel|feedback",
  "targetId": "2026-07-04-first-camping",
  "referrer": "nullable",
  "userAgentHash": "nullable",
  "metadata": {}
}
```

민감정보는 이벤트 원문에 저장하지 않는다. IP, user agent, 인증 계정 정보가 필요한 경우에는 서버 측에서 해시 또는 별도 보안 저장소로 분리하고, 공개 리포트에는 집계값만 사용한다.

### 집계 데이터

이벤트 원문은 운영 분석용으로 보관하고, 화면과 스코어링에는 집계 테이블을 사용한다.

| 집계 | 대상 | 예시 지표 |
|---|---|---|
| 작가 집계 | 작가 | 작가 페이지 조회, 작가 피드백 수, 관심 작품 전환 |
| 작품 집계 | 작품 | 조회, 관심, 재방문, 작품 피드백 |
| 회차 집계 | 회차 | 조회, 완독률, 좋아요, 회차 피드백 |
| 사용자 신뢰 집계 | 사용자 | 피드백 채택률, 신고율, 반복 참여 |

## 목표 설정과 달성 판정

목표는 단순한 홍보 문구가 아니라 운영 판단 기준이다. 작가, 작품, 회차별로 목표를 만들고, 백그라운드 이벤트와 피드백 집계를 통해 달성률을 갱신한다.

### 목표 단위

| 단위 | 목표 예시 | 주요 데이터 |
|---|---|---|
| 작가 | 인증 회원 작가 피드백 10건, 신고율 기준 이하 유지 | 작가 페이지 조회, 작가 피드백, 피드백 반응, 신고 |
| 작품 | 관심 등록 100명, 재방문 30명 | 작품 조회, 관심 등록, 재방문, 작품 피드백 |
| 회차 | 완독률 70%, 다음 회차 이동률 기준 달성 | 회차 조회, 컷 노출, 완독, 다음 회차 클릭 |

### 목표 데이터 필드

```json
{
  "goalId": "goal-series-first-readers",
  "targetType": "author|series|episode",
  "targetId": "bd-crew-camping-log",
  "metric": "favorite_added",
  "currentValue": 76,
  "targetValue": 100,
  "status": "draft|active|achieved|missed|paused",
  "startsAt": "2026-07-07",
  "endsAt": "2026-08-01",
  "achievementPolicy": "badge|score_boost|editorial_review"
}
```

### 달성 판정 원칙

- 목표는 운영자가 만들고 수정한다.
- 달성률은 원본 이벤트가 아니라 정제된 집계값으로 계산한다.
- 피드백 기여는 인증 회원과 신뢰 회원만 반영한다.
- 비정상 반복 조회, 신고 누적 데이터는 달성률에서 제외하거나 감산한다.
- 달성 결과는 공개 숫자와 내부 운영 점수로 분리한다.
- 달성 실패도 폐기가 아니라 개선 신호로 남긴다.

### 스코어링 연결

목표 달성은 스코어링의 입력 중 하나다. 예를 들어 1화 완독률 목표를 달성하면 회차 스코어와 다음 회차 제작 우선순위에 반영하고, 작가 피드백 목표를 달성하면 작가 신뢰도 보정에 사용한다. 목표가 조작 가능한 단일 지표에 의존하지 않도록 최소 두 개 이상의 이벤트 또는 피드백 신호를 함께 본다.

## 스코어링 설계

스코어링은 단일 별점이 아니라 여러 신호의 조합이다.

### 기본 신호

- 인증 회원 피드백
- 조회 수
- 완독률
- 재방문
- 관심 등록
- 좋아요 또는 공감
- 신고
- 운영자 검수

### 가중치 기준

초기 기준은 다음과 같이 둔다.

- 익명 피드백 가중치: `0`
- 인증 회원 피드백 가중치: `1.0`
- 신뢰 회원 피드백 가중치: `1.2` 이상 검토
- 신고가 많은 피드백은 대상 스코어에서 제외하거나 감점
- 운영자 검수 통과 피드백은 신뢰도 보정

### 대상별 스코어

| 스코어 | 주요 입력 |
|---|---|
| 작가 스코어 | 작가 피드백, 작가 페이지 재방문, 작품 관심 전환, 신고율 |
| 작품 스코어 | 작품 조회, 관심 등록, 작품 피드백, 회차 완독 평균 |
| 회차 스코어 | 회차 조회, 완독률, 회차 피드백, 다음 회차 이동률 |

스코어는 독자에게 바로 공개하는 숫자와 운영자가 보는 내부 점수를 분리할 수 있다. 공개 점수는 조작 가능성을 줄이기 위해 구간, 뱃지, 순위 등으로 표현하는 방식을 우선 검토한다.

## 운영 원칙

- 정적 프론트에는 공개 가능한 집계 데이터만 둔다.
- 인증, 피드백 저장, 조회 이벤트 수집은 별도 백엔드로 분리한다.
- 이미지 원본과 작가 업로드 자산은 Git 저장소나 DB가 아니라 object storage에 저장한다.
- 무료 베타는 사용량 관측을 전제로 하며, 무료 한도 70% 접근 시 유료 전환 여부를 결정한다.
- 작가 제출은 API/MCP/GPT Actions로 받을 수 있지만, 공개 전 검수 상태를 반드시 거친다.
- AI 저작도구 worker는 독자 서비스와 분리해 비용, 권한, 장애 범위를 격리한다.
- 피드백은 작성 즉시 공개하지 않고 신고/필터/운영자 검수 여지를 둔다.
- 스코어링은 원본 로그가 아니라 정제된 집계와 신뢰도 값을 기반으로 한다.
- production deployment, remote push, DB migration, secret rotation은 명시 승인 후 진행한다.
